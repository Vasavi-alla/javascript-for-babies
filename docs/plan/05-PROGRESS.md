## Standing user feedback (apply to ALL future work)
- **Metaphor weaning (2026-07-03).** Training-wheel metaphors must fade as the learner matures: Phases 0–1 metaphor-first ("boxes", "labels", "machines"); Phases 2–3 real term first with the metaphor in parentheses ("memory (the boxes)", "function (the machine)"); Phase 4+ real terms — metaphors only as occasional callbacks. Never introduce a new concept in Phase 4+ with a metaphor as its primary name.
- **Hands-on beats clicking (2026-07-03).** Writing real code that really runs is the core learning act; click-interactions support it, never replace it. **Exercises live INSIDE lessons** (`PlayExtra` CodeExercise; separate practice-bay pages were tried and removed — "very less and at the very last"). Not every lesson needs one; place each where its concept lands, cumulative where possible (functions exercises re-test let/const/if/loops).
- **Fun-fact standard (2026-07-03).** A fun fact must have an everyday-world hook — V8 named after the car engine, "Java is to JavaScript as ham is to hamster", AT&T's 9-hour collapse from one break. History trivia (person + year + language) reads as "a fact, not fun" and gets replaced: Eich-10-days was swapped out; 3.3 Dartmouth-BASIC and 3.4 CoffeeScript are flagged for the same treatment in the 3.3–3.5 refresh.
- **Teach first, then work (2026-07-03).** A mission's first step is KT — explain the syntax/anatomy before asking the learner to build anything.
- **Typed checks, silent steppers (2026-07-03).** "Watch it happen" never asks questions (prediction gates removed engine-wide; the `Step.prediction` field is dead). Lesson checks are typed-input (`kind: 'type-output'`, exact answers + accepted variants) wherever a concrete value/word/count exists; MCQ only where the options are the teaching (e.g. 0.1+0.2 WHY, var-tutorial judgment, L01 metaphors).
- **Exercises must be TRANSFER, never clones (2026-07-03).** An in-lesson exercise may not re-skin the watched code (watch greet → write welcome was rejected). Same concept, different surface, PLUS at least one new twist the stepper didn't show (a loop in the body, a computation instead of gluing, two calls to one function, two callbacks, values combined with +…). Applied: 3.1 cheer, 3.2 weather-order-trap, 3.3 half(10)+half(6), 3.4 arrow+ternary bigger, 3.6 chorus-twice, 3.8 fullDay(two callbacks), 2.1 pass/fail with >=.
- **Exercise wording = LeetCode style (2026-07-03).** Story → precise requirements (exact names, exact output, constraints — the WHAT) → exact expected output block. Never step-by-step code instructions (the HOW is the learner's job; inspection labels and show-answer teach it).
- **No prose walls in notes (2026-07-03).** Shop notes / under-the-hood as scannable sticky-note cards with bold headers, not paragraphs.
- **User's example phrasings are direction, not copy** — rewrite in the app's own voice.
- **Branding rule (2026-07-03):** app-level identity = the notebook/sketchbook ("your notebook", "this notebook keeps your page"). Phase worlds (Machine Shop, Warehouse…) are per-phase costumes and must never leak into app-level copy (home page, break coach, stickers, TeachBack shell).
- **Interaction clarity rules (2026-07-03, from 3.1 review):** (a) clickable things must look unmistakably clickable — never low-opacity tints on cream paper; (b) every challenge = short prompt + ONE highlighted "👉 do this" line (people skim); (c) animations must show the actual work happening (assembly line gluing the sentence), never abstract stand-ins like gears; left-to-right flow reads best; (d) ALL interactive progress persists — challenges, exercises — revisits resume, never restart.
- **Write for true newbies.** No expert one-liners anywhere in the UI; every phase/lesson listing carries a plain-words explanation. Phase intros live in `src/content/phase-intros.ts`.
- **Motivation before mechanism.** Every phase (and lesson) opens with "why do we need this?" — what breaks without the concept (`whyNeeded` in phase-intros).
- **Foundations get priority.** Phases 0–1 are visually elevated ("start here — the foundation").
- **Name and explain runtime concepts explicitly** — synchronous, asynchronous, blocking, non-blocking, single-threaded, Node.js's model (Phase 6 + 8 intros do this; lessons must too).
- Structure: compact home map → rich per-phase pages, not everything on one page.
- **Labs are per phase** (`src/labs/`, rendered on phase pages). `/design` is only the app's internal style guide.
- **Prediction/quiz wording must stand alone** in plain language — never depend on a metaphor from an earlier step (e.g. say "keeps age but swaps what it remembers", not "the label moves to a brand-new box").
- **Keep "Under the Hood" compact** (user request 2026-07-03): 3 short paragraphs max (~250 words total incl. the fun fact), matching Phase 0–2 lessons. Long walls of text lose the learner's attention. See memory: feedback-compact-under-the-hood.
